## Build & Dev Commands

- `npm run build` ‚Äî Compile TypeScript (`tsc`) to `dist/`
- `npm run dev` ‚Äî Run server in dev mode via `tsx` (no build needed)
- `npm start` ‚Äî Run compiled server from `dist/index.js`
- `npm run generate` ‚Äî Regenerate `generated/docs-manifest.json` and `generated/method-registry.ts` from Kotlin KDoc + TS type declarations
- `npm run prepublishOnly` ‚Äî Generate + build (runs before `npm publish`)

The `generate` step requires the parent `cardinal-sdk` repo and `@icure/cardinal-sdk` installed in `node_modules`. It reads Kotlin source files from `../cardinal-sdk/src/commonMain/` and TypeScript declarations from `node_modules/@icure/cardinal-sdk/`.

## Architecture

This is an MCP (Model Context Protocol) server that exposes Cardinal SDK documentation and operational tools over stdio transport. It allows AI assistants to search Cardinal SDK docs and execute SDK operations.

### Entry Flow
- `src/index.ts` ‚Äî CLI entry point, creates server and connects stdio transport
- `src/server.ts` ‚Äî Creates `McpServer`, registers all tools and resources

### Tools (SDK Operations)
Tools require SDK initialization via `cardinal_init` before use. All operational tools delegate to `src/tools/dispatch.ts`.

- **`cardinal_init`** (`src/tools/init.ts`) ‚Äî Initializes `CardinalSdk` with credentials, stores instance in `sdk-state.ts` singleton
- **`cardinal_admin`** (`src/tools/admin-tools.ts`) ‚Äî Group, User, Role, Permission, System, Auth APIs
- **`cardinal_data_owner`** (`src/tools/data-owner-tools.ts`) ‚Äî HealthcareParty, Patient, Device APIs with flavour support (decrypted/encrypted/tryAndRecover)
- **`cardinal_crypto`** (`src/tools/crypto-tools.ts`) ‚Äî Crypto, Recovery, ShamirKeysManager, DataOwner, CardinalMaintenanceTask APIs
- **`search_documentation`** (`src/resources/search.ts`) ‚Äî Full-text search across docs manifest (no SDK init required)

### Dispatch System (`src/tools/dispatch.ts`)
Generic method dispatch: resolves API object from SDK instance, looks up method in the generated registry (`generated/method-registry.ts`), builds arguments (including filter and model construction), calls method, and serializes the result. Handles `PaginatedListIterator` results via `src/helpers/pagination.ts`.

### Resources (Documentation)
MCP resources serve Cardinal SDK documentation from `generated/docs-manifest.json`:
- `src/resources/overview.ts` ‚Äî SDK overview
- `src/resources/api-docs.ts` ‚Äî API method documentation (`cardinal://docs/api/{name}`)
- `src/resources/model-docs.ts` ‚Äî Model/entity documentation (`cardinal://docs/model/{name}`)
- `src/resources/filter-docs.ts` ‚Äî Filter documentation (`cardinal://docs/filter/{name}`)
- `src/resources/tutorial-docs.ts` ‚Äî Tutorial content (`cardinal://docs/tutorial/{slug}`)

### Generated Files (`generated/`)
Do not manually edit. Regenerate with `npm run generate`.
- `docs-manifest.json` ‚Äî All documentation data (APIs, models, filters, tutorials) extracted by `scripts/extract-docs.ts`
- `method-registry.ts` ‚Äî API method signatures and metadata for runtime dispatch, generated by `scripts/generate-registry.ts`

### Helpers (`src/helpers/`)
- `docs-loader.ts` ‚Äî Loads and caches `docs-manifest.json`
- `filter-builder.ts` ‚Äî Constructs Cardinal SDK filter objects from plain JSON
- `model-builder.ts` ‚Äî Constructs Cardinal SDK model instances from plain JSON
- `pagination.ts` ‚Äî Collects first page from `PaginatedListIterator`, stores iterator for continuation
- `serialization.ts` ‚Äî Formats SDK results as JSON text for MCP responses
